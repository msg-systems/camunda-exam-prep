Free, open-source, static study tool for the **Camunda 8 Certified Developer (C8-CP-DV)** exam.
Drill against a realistic pool of scenario-style questions with per-option scoring, full review walkthrough, and a mock-exam mode that mirrors the real blueprint (60 questions / 75 minutes / 65 % to pass).

> **Not affiliated with or endorsed by Camunda Services GmbH.** All questions are original, written for personal study, and reference public Camunda 8 documentation. "Camunda" is a trademark of its respective owner. See [LICENSE](LICENSE).

---

## Highlights

- **840 scenario questions** across 8 topics, sourced from 14 hand-authored Markdown sets and parsed into JSON.
- **Per-option scoring (1–10)** with verdict + explanation rendered under every option on reveal.
- **Deterministic option shuffle** — no "answer A is always correct" cheat; same question gets the same shuffle across re-imports so progress is stable.
- **Mock exam mode** — 60 weighted questions, 75-minute timer, navigator, flag-for-review, confirmation modal before submit, full post-exam walkthrough with wrong-only filter.
- **Practice mode** — topic-by-topic, instant feedback, doc links per question.
- **Review mode** — re-study only the questions you got wrong or bookmarked.
- **Dashboard** — coverage per topic, recent attempts, scores.
- **localStorage persistence** — no backend, no telemetry.

## Stack

React 18 · TypeScript · Vite · Tailwind · Zustand · Vitest. Static SPA, deployable to any static host (GitHub Pages, Netlify, S3, plain nginx).

## How the question pool is built

The pool is **not hand-written JSON** — it's generated from Markdown source files by [`pipeline/import-scenarios.mjs`](pipeline/import-scenarios.mjs).

### Source format

Each set is a single Markdown file vendored inside the repo at [`pipeline/sources/`](pipeline/sources/):

- `pipeline/sources/set01.md`
- `pipeline/sources/set02.md` … `pipeline/sources/setNN.md`

The importer auto-discovers every `setNN.md` in that folder — no code changes needed when you add or remove a set.

Each file contains 60 questions following a strict template:

```md
## Question 1: <Topic Name> (Weighting: 15%)

**Scenario:** ...

**<Question text>?**

- **a)** Option text... Documentation: [Title](https://...)
- **b)** ...
- **c)** ...
- **d)** ...

**🔍 Explanations & Correct Answer**

- **Option a) — Correct.** Detailed explanation.
- **Option b) — Wrong reason.** ...
- **Option c) — Wrong reason.** ...
- **Option d) — Wrong reason.** ...

**Per-option scoring (1–10):**
- **a) 10/10** — short note.
- **b) 3/10** — short note.
- **c) 2/10** — short note.
- **d) 4/10** — short note.

**Correct Answer:** ...
**Official Documentation Link:** https://docs.camunda.io/...
```

### What the pipeline does

For every question, it:

1. Parses scenario / question / 4 options / per-option explanations / per-option 1–10 scoring / docs link.
2. Strips any leftover **Bulgarian (Cyrillic) text** from author notes.
3. **Shuffles options deterministically** (FNV-1a hash of the question id → mulberry32 PRNG → Fisher–Yates) so:
   - The correct answer isn't always `a)` (source MDs had ~94 % bias to A).
   - Re-running the importer produces the **same shuffle** for the same id, so user progress tied to question ids stays valid.
4. Writes everything to `src/data/questions/scenarios-imported.json`.

### Adding a new set of 60 questions

1. **Drop the MD file** into `pipeline/sources/` using the naming pattern `setNN.md` (e.g. `set12.md`). The number must be unique — it's baked into question ids (`imp-s12-mod-01`).
2. **Run the importer:**
   ```bash
   node pipeline/import-scenarios.mjs
   ```
   Look for `Set 12 (set12.md): parsed 60, skipped 0`. Anything skipped is printed with a reason — usually a malformed section header.
3. **Validate + restart:**
   ```bash
   npm test          # expect 7/7
   npm run build     # expect ✓
   npm run dev       # reload browser
   ```

If a new MD introduces a topic name not in `TOPIC_MAP` inside the pipeline, add the mapping there.

### Removing a set

Delete the file from `pipeline/sources/` and re-run the importer. Done.

## Question schema

Each entry in `scenarios-imported.json` matches the `Question` type in [`src/types/index.ts`](src/types/index.ts):

```jsonc
{
  "id": "imp-s1-mod-01",
  "topic": "modeling",
  "subtopic": "Modeling",
  "difficulty": "hard",
  "style": "scenario",
  "scenario": "...",
  "question": "...",
  "options": [
    { "id": "a", "text": "..." },
    { "id": "b", "text": "..." },
    { "id": "c", "text": "..." },
    { "id": "d", "text": "..." }
  ],
  "correctOptionId": "b",
  "explanation": "Aggregated explanation block.",
  "optionExplanations": {
    "a": { "verdict": "Wrong reason", "text": "...", "score": 3 },
    "b": { "verdict": "Correct", "text": "...", "score": 10 },
    "c": { "verdict": "Wrong reason", "text": "...", "score": 2 },
    "d": { "verdict": "Wrong reason", "text": "...", "score": 4 }
  },
  "docs": [{ "title": "Camunda 8 documentation", "url": "https://docs.camunda.io/..." }]
}
```

Topic ids and exam weights live in [`src/data/topics.ts`](src/data/topics.ts) and match the official blueprint (Modeling 15 %, Configuring Processes 22 %, Decisions & DMN 11 %, Forms 5 %, Connectors 6 %, Extensions & Integrations 25 %, Managing Development 15 %, Dev Environment 1 %).

## Tests

`npm test` runs the schema validator:

- All question ids are unique.
- Every question has exactly 4 options with ids `a`/`b`/`c`/`d`.
- `correctOptionId` resolves to one of the options.
- Every doc URL is well-formed.
- Topic weights sum to 100.

## Making this a git repo (today)

The project ships with a sensible `.gitignore` (`node_modules`, `dist`, `.DS_Store`, `*.log`, `.vite`, `coverage`). To initialise:

```bash
cd ~/camunda-exam-prep
git init
git config user.name  "Your Name"        # one-time, or use --global
git config user.email "you@example.com"  # required — commit fails (exit 128) without it
git add .
git commit -m "Initial commit: Camunda 8 exam prep, 840 questions, 14 sets"
```

**Optional — push to GitHub:**

```bash
# create an empty repo on github.com first, then:
git branch -M main
git remote add origin git@github.com:<you>/camunda-exam-prep.git
git push -u origin main
```

### Deploying to GitHub Pages

This repo includes [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml). After your first push to `main`:

1. On GitHub: **Settings → Pages → Build and deployment → Source: GitHub Actions**.
2. Push to `main` (or run the workflow manually) — the action runs tests, builds, and publishes `dist/` to Pages.
3. Your site goes live at `https://<you>.github.io/<repo>/`.

The Vite config uses `base: './'` and the app uses HashRouter, so it works on any subpath without further config.

A second workflow ([`.github/workflows/ci.yml`](.github/workflows/ci.yml)) runs lint + tests + build on every PR.

### Things to consider before pushing publicly

- The Markdown source sets are vendored in `pipeline/sources/` so the importer runs identically on macOS, Linux, and Windows with no path edits.
- `src/data/questions/scenarios-imported.json` is committed so the app works without running the pipeline. Re-run `node pipeline/import-scenarios.mjs` after editing any `setNN.md`.
- No secrets, no telemetry, no analytics — safe to make public as-is.
- See [CONTRIBUTING.md](CONTRIBUTING.md) for the contribution flow.
